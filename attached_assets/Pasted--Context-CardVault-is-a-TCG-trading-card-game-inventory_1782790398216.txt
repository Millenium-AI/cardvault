
## Context

CardVault is a TCG (trading card game) inventory management app built with:
- **Frontend:** React + TypeScript + TanStack Query (`@tanstack/react-query`)
- **Backend:** Express + TypeScript
- **Database:** Supabase (Postgres)
- **File:** `server/routes.ts` is ~56KB and contains all API route logic

There are **3 confirmed bugs** that need to be fixed. Instructions below are exact — read them fully before touching any code.

---

## Bug 1 — All Cards Get Wrong Game From Multi-Game CSV Upload

### What's wrong
When a user uploads a CSV that contains multiple games (e.g. Dragon Ball, Sorcery, Pokémon all in one file), every single card gets stamped with the **same game** — whatever the dropdown was set to or whatever game was detected from row 1. A 5,000-row mixed CSV results in all 5,000 cards tagged as `"pokemon"` (or whichever game was first).

### Why it happens (two gaps)

**Gap A — Client sends one game for the whole file**

In `client/src/pages/Uploads.tsx`, the function `detectGameFromFile(file)` reads only the **first data row** of the CSV and maps it to a slug. That single slug is then set on component state (`setGame(slug)`) and sent as one `game` field in the `FormData` for the entire upload.

**Gap B — Server ignores `Product Line` column per row**

In `server/routes.ts`, the function `mapCsvRow(raw, game, rowIndex, uploadId)` already extracts `sourceProductLine` from each CSV row:
```ts
const sourceProductLine = k("Product Line", "product_line", "Game") || null;
```
But it **never uses it to determine the game**. The `game` parameter (upload-level, same for all rows) is passed directly into `buildMatchKey` and is NOT stored on the returned row object at all.

Then in the approve handler (`POST /api/uploads/:id/approve`):
```ts
const uploadRecord = await storage.getUpload(userId, uploadId);
const game = uploadRecord?.game || "pokemon";
// then inside rpcNewItems.map():
return { game, ... } // same game for every card
```

### Fix Instructions

#### Step 1 — Add `detectGameFromProductLine()` to `server/routes.ts`

Place this function **directly above** the existing `mapCsvRow` function:

```ts
function detectGameFromProductLine(productLine: string | null, fallback: string): string {
if (!productLine) return fallback;
const pl = productLine.toLowerCase();
if (pl.includes("pokemon") || pl.includes("pokémon")) {
if (pl.includes("japan") || pl.includes(" jp") || pl.includes("(jp)")) return "pokemon-jp";
return "pokemon";
}
if (pl.includes("one piece")) return "one-piece";
if (pl.includes("sorcery")) return "sorcery";
if (pl.includes("dragon ball")) return "dragon-ball";
if (pl.includes("magic") || pl.includes("the gathering") || pl === "mtg") return "mtg";
if (pl.includes("star wars")) return "star-wars";
if (pl.includes("lorcana")) return "lorcana";
if (pl.includes("yu-gi-oh") || pl.includes("yugioh")) return "yugioh";
if (pl.includes("digimon")) return "digimon";
if (pl.includes("flesh and blood") || pl.includes("flesh & blood")) return "fab";
return fallback;
}
```

#### Step 2 — Modify `mapCsvRow()` in `server/routes.ts`

Inside the body of `mapCsvRow`, after the line that extracts `sourceProductLine`:
```ts
const sourceProductLine = k("Product Line", "product_line", "Game") || null;
```

Add immediately after:
```ts
const resolvedGame = detectGameFromProductLine(sourceProductLine, game);
```

Then update **two things** in the return object:

1. Change `buildMatchKey(...)` to use `resolvedGame` instead of `game`:
```ts
// BEFORE
normalizedMatchKey: buildMatchKey(productName, number, condition, sourcePrinting, sourceSetName, game),

// AFTER
normalizedMatchKey: buildMatchKey(productName, number, condition, sourcePrinting, sourceSetName, resolvedGame),
```

2. Add `game: resolvedGame` as a field in the returned object (it was missing entirely before):
```ts
return {
id: crypto.randomUUID(),
uploadId,
rowIndex,
game: resolvedGame, // ← ADD THIS LINE
productName: productName || "(unknown)",
// ... rest of existing fields unchanged ...
};
```

#### Step 3 — Modify the approve handler in `server/routes.ts`

Find the `POST /api/uploads/:id/approve` handler. Near the top of the handler body, find:
```ts
const uploadRecord = await storage.getUpload(userId, uploadId);
const game = uploadRecord?.game || "pokemon";
```

Change to:
```ts
const uploadRecord = await storage.getUpload(userId, uploadId);
const uploadLevelGame = uploadRecord?.game || "pokemon";
```

Then find the `rpcNewItems` mapping block. Inside the `.map((row: any) => {` callback, it currently does:
```ts
const parsed = parsedById.get(row.id);
// ...
return {
// ...
game, // ← single upload-level game for all rows
```

Change that to use per-row game with upload-level fallback:
```ts
const parsed = parsedById.get(row.id);
const resolvedGame = (parsed as any)?.game || uploadLevelGame;
// ...
return {
// ...
game: resolvedGame, // ← per-row game
```

> **Note:** Any other reference to the old `game` variable in the approve handler (e.g. passed to `refreshExistingInventoryPrices`) can remain using `uploadLevelGame` since that call operates on the whole upload.

#### Step 4 — Verify `server/storage.ts` passes `game` through `createParsedRows`

Open `server/storage.ts` and find the `createParsedRows` function. Confirm that the `game` field from each row object is included in the Supabase insert payload. If it's being stripped or not mapped to a column, add it.

If the `parsed_rows` table in Supabase does not have a `game` column, run this migration:
```sql
ALTER TABLE parsed_rows ADD COLUMN IF NOT EXISTS game text;
```

## Bug 2 — JustTCG API Is Called But Prices Are Never Saved

### What's wrong
After approving an upload, `enrichNewItemsWithLivePrices` is triggered via `setImmediate`. It successfully calls the JustTCG API and gets price data back. But then it tries to save with the **wrong Supabase filter column**, so zero rows ever get updated. The prices are fetched and silently discarded.

### Exact location — `server/routes.ts`

**In `enrichNewItemsWithLivePrices`** (the function triggered after approve), find this Supabase update:
```ts
await supabaseAdmin
.from("inventory_items")
.update({
current_raw_market_price: priceResult.price,
current_rounded_print_price: Math.ceil(priceResult.price),
price_last_fetched_at: new Date().toISOString(),
price_change_24hr: priceResult.priceChange24hr,
price_change_7d: priceResult.priceChange7d,
justtcg_card_uuid: priceResult.cardUuid,
justtcg_variant_uuid: priceResult.variantUuid,
})
.eq("inventory_item_id", item.id) // ← BUG: this column does not exist
.eq("user_id", userId);
```

The column is `"id"` not `"inventory_item_id"`. The correct filter used elsewhere in the same file (e.g. in `refreshExistingInventoryPrices`) is `.eq("id", item.id)`.

**In `POST /api/prices/refresh`** (the manual price refresh endpoint), find the identical bug:
```ts
await supabaseAdmin
.from("inventory_items")
.update({ ... })
.eq("inventory_item_id", item.id) // ← SAME BUG
.eq("user_id", userId);
```

### Fix Instructions

**Fix A** — In `enrichNewItemsWithLivePrices`, change:
```ts
.eq("inventory_item_id", item.id)
```
to:
```ts
.eq("id", item.id)
```

**Fix B** — In `POST /api/prices/refresh`, change:
```ts
.eq("inventory_item_id", item.id)
```
to:
```ts
.eq("id", item.id)
```

That is the complete fix. One word changed in two places.

### While you are there — also verify
The `enrichNewItemsWithLivePrices` function filters items by `i.sourceTcgplayerId`. Confirm that `storage.listInventoryItems()` in `server/storage.ts` maps the DB column `source_tcgplayer_id` to the camelCase key `sourceTcgplayerId` on the returned object. If this mapping is missing, the `newItems` array will always be empty and JustTCG is never called at all (separate from the `.eq` bug).

## Bug 3 — Inventory Doesn\'t Refresh After Approve or Page Navigation

### What's wrong
After approving an upload, the Inventory page still shows old data. The user has to manually hard-refresh the browser to see newly added cards. The same issue occurs when navigating away and back — stale cached data is shown.

### Exact location — `client/src/pages/Uploads.tsx`

Find the `approveMut` mutation (created with `useMutation`). In its `onSuccess` callback, the current invalidations are:
```ts
onSuccess: () => {
queryClient.invalidateQueries({ queryKey: ["/api/uploads"] });
queryClient.invalidateQueries({ queryKey: ["/api/dashboard/stats"] });
queryClient.invalidateQueries({ queryKey: ["/api/labels/new"] });
toast({ title: "Merge approved", description: "Inventory updated successfully." });
onDone();
},
```

`/api/inventory` is never invalidated, so the Inventory page cache is never cleared.

### Fix A — Add inventory invalidation in `client/src/pages/Uploads.tsx`

```ts
onSuccess: () => {
queryClient.invalidateQueries({ queryKey: ["/api/uploads"] });
queryClient.invalidateQueries({ queryKey: ["/api/dashboard/stats"] });
queryClient.invalidateQueries({ queryKey: ["/api/labels/new"] });
queryClient.invalidateQueries({ queryKey: ["/api/inventory"] }); // ← ADD THIS
toast({ title: "Merge approved", description: "Inventory updated successfully." });
onDone();
},
```

### Fix B — Force fresh data on mount/focus in `client/src/pages/Inventory.tsx`

Find the main `useQuery` that fetches inventory items. Add these three options so the page always fetches fresh data when navigated to or when the tab is focused:

```ts
const { data: inventoryItems } = useQuery({
queryKey: ["/api/inventory", { game, condition, status, search }],
queryFn: () => apiRequest(...).then(r => r.json()),
staleTime: 0, // ← ADD: treat cache as immediately stale
refetchOnMount: true, // ← ADD: refetch every time component mounts
refetchOnWindowFocus: true, // ← ADD: refetch when user returns to tab
});
```
